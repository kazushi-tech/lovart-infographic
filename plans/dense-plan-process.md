# 濃密プラン作成プロセス（Dense Plan Process）

## 目的

実装前のプランを「そのまま機械的に実装しても壊れない」レベルまで濃密にする。
codex-review 単体では防げなかった品質問題を、**多段レビュー + 具体性の強制** で解決する。

---

## 前提：なぜ必要か

### 従来の失敗パターン
1. プランが抽象的 →「〇〇を修正する」だけで、具体的な変更行が不明
2. codex-review が通る → だが「実装したら動くか」は検証していない
3. 一括実装 → 5マイルストーンを一気にやって何が壊れたか特定不能
4. 外部レビューなし → 同じAIが書いて同じAIがレビューするので盲点が残る

### この仕組みで防ぐこと
- **抽象的な計画を許さない** — 全変更箇所にfile:line参照を強制
- **単一AI の盲点を排除** — Codex（隣カラム）に外部レビューさせる
- **壊れうるシナリオを事前列挙** — 「何が壊れうるか」を計画に含める
- **1マイルストーンごとの検証基準** — 各ステップの完了条件を明文化

---

## フロー全体像

```
Step 1: 現状スナップショット
    ↓
Step 2: プラン初稿（Claude Code）
    ↓
Step 3: 内部レビュー（codex-review）
    ↓
Step 4: 外部AIレビュー（Codex / ChatGPT）
    ↓
Step 5: 差分統合 & 濃密化
    ↓
Step 6: ユーザー最終承認
    ↓
Step 7: 実装（1マイルストーンずつ）
```

---

## Step 1: 現状スナップショット

**目的**: 変更対象のコードを正確に把握し、プランに根拠を持たせる。

### やること
- 変更対象ファイルを全て読む
- 各ファイルの**現在の行数・主要関数・export一覧**を記録する
- 現在の動作フロー（ユーザーが操作する順序）をテキストで書き出す
- `npm run lint` と `npm run build` の結果を記録する（ベースライン）

### 出力フォーマット
```markdown
## 現状スナップショット

### 変更対象ファイル
| ファイル | 行数 | 主要export | 役割 |
|---------|------|-----------|------|
| src/components/AppShell.tsx | 474行 | AppShell (default) | メインシェル、インタビュー+エディター |
| ... | ... | ... | ... |

### 現在の動作フロー
1. トップ画面 → 「テーマを入力する」or「サンプルで試す」
2. テーマ入力 → styleId選択 → slideCount → ...
3. 全回答後 → 「スライドを生成する」ボタン
4. 生成後 → 3カラムエディター

### ベースライン
- lint: PASS / FAIL (エラー内容)
- build: PASS / FAIL (警告内容)
```

---

## Step 2: プラン初稿（Claude Code）

**目的**: 具体的な変更内容を、コード差分レベルで記述する。

### 必須セクション

#### 2-A: マイルストーン分割
- 最大3ファイル変更/マイルストーン
- 各マイルストーンが独立してビルド・動作すること

#### 2-B: 変更箇所の具体記述（全マイルストーン共通）

各変更に対して以下を書く:

```markdown
### 変更: [説明]
- **ファイル**: `src/components/AppShell.tsx`
- **対象行**: L129-L191 (`handleSendMessage` 関数)
- **変更前の動作**: currentQuestionIndex で QUESTIONS 配列を順番に進める
- **変更後の動作**: getNextQuestion(flowState) で次の未回答質問を取得
- **差分イメージ**:
  ```diff
  - const currentQuestion = getCurrentQuestion(flowState);
  - if (!currentQuestion) {
  + const pendingQuestion = getNextQuestion(flowState);
  + if (!pendingQuestion) return;
  ```
- **影響範囲**: handleSendMessage を呼ぶ箇所 (ChatInterviewSidebar 経由)
- **壊れうるシナリオ**: テーマ入力が初回扱いにならない場合、flowState.answers にthemeが入らない
```

#### 2-C: 各マイルストーンの検証基準

```markdown
### M1 検証基準
- [ ] `npm run lint` PASS
- [ ] `npm run build` PASS
- [ ] ブラウザ確認: テーマ入力 → 次の質問に進む（ループしない）
- [ ] ブラウザ確認: 3問目まで進んで戻りボタンで2問目に戻れる
- [ ] Console: runtime error なし
```

#### 2-D: 触らないものリスト

明示的に「変更しない」ファイルや機能を列挙する。
スコープクリープを防ぐ。

```markdown
### 触らないもの
- `src/services/geminiService.ts` — 生成ロジックは変更しない
- `src/components/RightInspectorPanel.tsx` — エディター右パネルはそのまま
- エディターモードの3カラムレイアウト — 今回のスコープ外
```

---

## Step 3: 内部レビュー（codex-review）

**目的**: 構造・整合性の一次チェック。

### やること
- `/codex-review` をプラン初稿に対して実行
- Critical / Major の指摘を全て解消してから次へ進む

### 注意
- codex-review は**コードの存在確認**はするが、**実行時の動作保証**はしない
- 「コンパイルは通るが動作が壊れる」ケースは Step 4 で捕捉する

---

## Step 4: 外部AIレビュー（Codex / ChatGPT）

**目的**: Claude Code の盲点を別のAIで補完する。

### ユーザーの操作
1. Step 3 を通過したプランを **隣カラムの Codex にコピペ** する
2. 以下のプロンプトテンプレートで依頼する:

```
以下はReactアプリの変更計画書です。
実装者が見落としている致命的な問題を指摘してください。

特に注意してほしい点:
1. 変更前後でユーザーの操作フローが壊れないか
2. 型の不整合（TypeScript）
3. 状態管理の整合性（React useState の更新順序）
4. 既存の動作を意図せず破壊していないか
5. 変更箇所の具体性は十分か（曖昧な記述はないか）

[計画書をここに貼り付け]

[現在のAppShell.tsxのコードも貼り付け]
```

3. Codex の指摘を Claude Code に共有する

### Claude Code 側の対応
- ユーザーから共有された外部レビュー結果をプランに反映する
- 反論がある場合は根拠を示してユーザーに判断を委ねる

---

## Step 5: 差分統合 & 濃密化

**目的**: 内部 + 外部レビューの指摘を統合し、最終版プランを作成する。

### やること
- Step 3 (codex-review) の指摘を反映
- Step 4 (外部AI) の指摘を反映
- 全マイルストーンの検証基準を再確認
- 「壊れうるシナリオ」セクションを追加・更新

### 出力
- `plans/` 配下に最終版プランを保存
- ファイル名: `plans/{slug}-final.md`

---

## Step 6: ユーザー最終承認

**目的**: 実装着手の最終ゲート。

### Claude Code からユーザーへの確認事項
1. プランの全マイルストーンを提示
2. 各マイルストーンの変更ファイル数と検証基準を要約
3. 「このプランで実装を開始してよいか？」と明示的に確認

### ユーザーの判断
- 承認 → Step 7 へ
- 修正要求 → Step 2 に戻る

---

## Step 7: 実装（1マイルストーンずつ）

**目的**: プランに忠実に、検証しながら進める。

### ルール
1. **1マイルストーンごとに**:
   - コード変更
   - `npm run lint` + `npm run build`
   - ユーザー指定の方法で動作確認（CDP / Playwright / 手動）
   - 確認完了後、次のマイルストーンへ
2. **想定外の問題が発生したら**:
   - 実装を止める
   - ユーザーに報告して判断を仰ぐ
   - 勝手に回避策を取らない
3. **デプロイは全マイルストーン完了後**:
   - 全検証基準を満たしてからpush

---

## チェックリスト（プラン品質の自己評価）

プランを完成させる前に、以下を全て満たすか確認する:

- [ ] 全変更箇所に `file:line` 参照がある
- [ ] 全変更に差分イメージ（diff）がある
- [ ] 各マイルストーンが3ファイル以下の変更に収まっている
- [ ] 各マイルストーンに独立したビルド可能性がある
- [ ] 「壊れうるシナリオ」が各マイルストーンに1つ以上ある
- [ ] 「触らないものリスト」がある
- [ ] 検証基準に具体的な操作手順がある
- [ ] codex-review の Critical/Major が 0
- [ ] 外部AIレビューを通過している
- [ ] ユーザーの最終承認を得ている
