# Dense Plan — 濃密プラン作成スキル

「そのまま機械的に実装しても壊れない」レベルまでプランを濃密にする。
codex-review 単体では防げなかった品質問題を、**多段レビュー + 具体性の強制 + 外部AIレビュー** で解決する。

## 起源

2026-03-12 に5マイルストーンを一括実装して本番を壊した教訓から生まれたスキル。

## フロー

```
Step 1: 現状スナップショット
  ↓
Step 2: プラン初稿（diff付き）
  ↓
Step 3: codex-review（内部レビュー）
  ↓
Step 4: 外部AIレビュー（ユーザーがCodex等に依頼）
  ↓
Step 5: 差分統合 & 最終版プラン
  ↓
Step 6: ユーザー最終承認
  ↓
Step 7: 実装（1マイルストーンずつ、検証挟む）
```

---

## Step 1: 現状スナップショット

変更対象を正確に把握する。**コードを読まずにプランを書くことは禁止。**

### やること

1. 変更対象ファイルを **全て読む**
2. 以下のテーブルを作成:

```markdown
### 変更対象ファイル
| ファイル | 行数 | 主要export | 役割 |
|---------|------|-----------|------|
| src/components/AppShell.tsx | 474行 | AppShell (default) | メインシェル |
```

3. 現在のユーザー操作フローをテキストで書き出す:

```markdown
### 現在の動作フロー
1. トップ画面で「テーマを入力する」or「サンプルで試す」を選択
2. テーマ入力 → styleId選択 → slideCount → ...
```

4. ベースラインを記録:

```bash
npm run lint   # 結果を記録
npm run build  # 結果を記録
```

---

## Step 2: プラン初稿

### 絶対ルール

- **全変更箇所に `file:line` 参照をつける**
- **全変更に diff イメージをつける**
- **1マイルストーン = 最大3ファイル変更**
- **各マイルストーンが独立してビルド・動作すること**

### 各変更の記述フォーマット

`references/change-template.md` に従って書く。最低限以下を含む:

```markdown
### 変更: [説明]
- **ファイル**: `src/components/Foo.tsx`
- **対象行**: L42-L80 (`handleBar` 関数)
- **変更前の動作**: [具体的に]
- **変更後の動作**: [具体的に]
- **差分イメージ**:
  ```diff
  - oldCode();
  + newCode();
  ｀｀｀
- **影響範囲**: [この関数を呼ぶ箇所]
- **壊れうるシナリオ**: [具体的に]
```

### 各マイルストーンの検証基準

```markdown
### M1 検証基準
- [ ] `npm run lint` PASS
- [ ] `npm run build` PASS
- [ ] ブラウザ確認: [具体的な操作手順]
- [ ] Console: runtime error なし
```

### 触らないものリスト（必須）

スコープ外を明示的に宣言する:

```markdown
### 触らないもの
- `src/services/geminiService.ts` — 今回変更しない
- エディターモードのレイアウト — スコープ外
```

---

## Step 3: codex-review（内部レビュー）

プラン初稿に対して `/codex-review` を実行する。

- Mode: **Plan Gate**
- Critical / Major が **0** になるまで修正
- codex-review は構造・整合性チェック。実行時の動作保証はしない

---

## Step 4: 外部AIレビュー

**ユーザーが隣カラムの Codex（または ChatGPT 等）にレビューを依頼する。**

Claude Code は以下を出力してユーザーに渡す:

1. **レビュー依頼プロンプト**（`references/external-review-prompt.md` のテンプレート使用）
2. **プラン全文**
3. **変更対象ファイルの現在のコード**

ユーザーが外部AIの指摘を共有したら:
- 指摘を一つずつ確認し、プランに反映する
- 反論がある場合は根拠を示し、**ユーザーに判断を委ねる**（勝手に却下しない）

---

## Step 5: 差分統合 & 最終版プラン

Step 3 + Step 4 の指摘を統合し、`plans/{slug}-final.md` として保存する。

### 最終版プランの品質チェックリスト

以下を **全て** 満たすこと:

- [ ] 全変更箇所に `file:line` 参照がある
- [ ] 全変更に差分イメージ（diff）がある
- [ ] 各マイルストーンが3ファイル以下
- [ ] 各マイルストーンが独立してビルド可能
- [ ] 「壊れうるシナリオ」が各マイルストーンに1つ以上
- [ ] 「触らないものリスト」がある
- [ ] 検証基準に具体的な操作手順がある
- [ ] codex-review PASS済み
- [ ] 外部AIレビュー通過済み

---

## Step 6: ユーザー最終承認

ユーザーに以下を提示して明示的な承認を得る:

1. マイルストーン一覧（変更ファイル数と検証基準の要約）
2. 想定リスクのサマリ
3. 「このプランで実装を開始してよいか？」

承認なしに実装に入ることは **禁止**。

---

## Step 7: 実装（1マイルストーンずつ）

### ルール

1. **1マイルストーン完了ごとに**:
   - `npm run lint` + `npm run build`
   - ユーザー指定の方法で動作確認（`/devtools-verify`、Playwright、手動等）
   - **確認完了を報告し、次のマイルストーンに進んでよいか確認**
2. **想定外の問題が発生したら**:
   - 実装を **止める**
   - ユーザーに報告し判断を仰ぐ
   - 勝手に回避策を取らない
3. **全マイルストーン完了後にデプロイ**:
   - 全検証基準を満たしてから push

### 検証方法の優先順位

ユーザーが指定した方法 > `/devtools-verify`（CDP） > Playwright > 手動

ユーザーの指定がある場合、**別の方法で代替しない**。

---

## アンチパターン（絶対にやらないこと）

1. **コードを読まずにプランを書く**
2. **diff なしで「〇〇を修正する」と書く**
3. **複数マイルストーンを一括実装する**
4. **ユーザー指定の検証方法を別のツールで代替する**
5. **外部AIレビューをスキップする**
6. **承認前に実装を始める**
7. **想定外の問題を勝手に回避する**
